# API TESTING FRAMEWORK – COMPLETE FUNCTIONAL OVERVIEW

## 1. Purpose
The API Testing Framework is a Java‑based automation tool for testing REST APIs. It uses Cucumber (BDD), Rest Assured, and Serenity BDD to provide:
- Declarative test definition via YAML files.
- Built‑in support for authentication, authorization, and functional tests.
- Dynamic header generation (pre‑script) from external services (JSON, text, XML).
- Automatic generation of test data (emails, UUIDs, timestamps, etc.).
- Detailed reports in Markdown, Cucumber HTML, and Serenity HTML.

## 2. Technology Stack
- **Java 21** – core language.
- **Maven** – build and dependency management.
- **Cucumber** – BDD framework with Gherkin syntax.
- **Rest Assured** – HTTP client for API calls.
- **Serenity BDD** – reporting and test orchestration.
- **Jackson** – YAML parsing.
- **JavaFaker** – random data generation.
- **JUnit 5** – test execution platform.

## 3. Key Features

### 3.1 Test Categorisation
Tests are organised into three main categories:
- **Authentication** – validates required headers (e.g., X‑API‑Key, X‑Transaction‑ID, X‑Timestamp) and their formats/values.
- **Authorization** – checks role‑based access control, endpoint permissions, and HTTP method allowances.
- **Functional** – covers CRUD operations, business logic validations, and end‑to‑end flows.

### 3.2 Declarative Test Definition (YAML)
Each test case is described in a YAML file. Example:
```yaml
tests:
  - name: "Create user with complete data"
    method: "POST"
    endpoint: "/api/v1/usuarios"
    body:
      content:
        nombre: "María González"
        email: "maria@ejemplo.com"
        edad: 28
    expected:
      status: 201
    extract:
      - name: "usuario_id"
        from: "$.id"
3.3 Pre‑script (Dynamic Headers)
Before a test runs, the framework can call an external service to obtain a header value. Supported response formats:

JSON – extract using JSONPath (extract_from).

XML – extract using XPath.

Plain text – extract using a regular expression (extract_regex) or use the whole body.

Configuration in base-headers.yaml:

yaml
pre_script:
  enabled: true
  headers:
    X-Auth-Token:
      service_url: "https://auth.service.com/token"
      method: "POST"
      body:
        grant_type: "client_credentials"
      extract_from: "$.access_token"
      response_type: "json"
      timeout: 5000
      retry: 3
3.4 Dynamic Value Generation
The framework can generate test data on the fly using placeholders like ${dynamic.email} or ${random.full_name}. Supported dynamic types:

email – unique email with timestamp.

uuid – random UUID.

timestamp – ISO 8601 timestamp.

random – numbers, strings, booleans, names, etc.

3.5 Reporting
Three report formats are automatically generated:

Markdown (target/reports/test-report-*.md) – detailed scenario breakdown with request/response.

Cucumber HTML (target/cucumber-reports/index.html) – visual execution report.

Serenity HTML (target/site/serenity/index.html) – professional dashboard with graphs.

5. Installation & Configuration
5.1 Prerequisites
Java 21

Maven 3.6+

Git (optional)

5.2 Steps
Clone the repository.

Edit src/test/resources/config.env with your API base URL, API key, and test credentials.

Edit src/test/resources/headers/base-headers.yaml to define static headers and pre‑script services.

Run mvn clean compile to build.

Execute a smoke test: mvn test -Dcucumber.filter.tags="@Smoke".

6. Running Tests
6.1 Common Maven Commands
Command	Description
mvn clean test	Run all tests.
mvn test -Dcucumber.filter.tags="@Functional"	Run functional tests.
mvn test -Dtest=FunctionalTestRunner	Run using a specific runner.
mvn test -Dcucumber.filter.name="Create user"	Run scenario by name.
mvn test -Denvironment=qa	Use a specific environment profile (requires separate config.env).
mvn test -Dcucumber.execution.dry-run=true	Validate steps without executing.
6.2 Parallel Execution
Enable parallelism with:

bash
mvn test -Dcucumber.execution.parallel.enabled=true -Dcucumber.execution.parallel.config.fixed.parallelism=4
6.3 Re‑running Failed Tests
After a run, failed tests are listed in target/rerun.txt. Re‑run them with:

bash
mvn test -Dcucumber.features=@target/rerun.txt
7. Generating Reports
Markdown: Generated automatically after mvn test. Located in target/reports/.

Serenity: mvn serenity:aggregate creates target/site/serenity/index.html.

All reports: mvn clean verify.

8. Extending the Framework
8.1 Adding a New Dynamic Value Type
Modify DynamicValueGenerator.java and add a new case in the generateByType or generateRandom methods.

8.2 Adding a New Header Type in Pre‑script
Edit base-headers.yaml under pre_script.headers. Define the service URL, method, body, extraction method, and response type.

8.3 Adding a New Test Category
Create a new folder under features/ and tests/.

Create a new runner class extending org.junit.platform.suite.api.Suite.

Add step definitions if new custom steps are needed.

9. Troubleshooting
Issue	Solution
java -version shows wrong version	Set JAVA_HOME to JDK 21 path.
mvn: command not found	Install Maven.
YAML files not found	Ensure they are under src/test/resources/ and the path in the .feature file matches.
Cannot resolve symbol 'En'	Use annotation‑based step definitions instead of lambda‑style.
socketTimeout method not found	Use .timeout() in Rest Assured (or .socketTimeout() only if you have Rest Assured 5.x).
Connection timeout	Increase timeout in base-headers.yaml for the relevant service.
10. Example: Writing a New Test
Create a YAML suite in src/test/resources/tests/my‑suite.yaml:

yaml
suite:
  name: "My new tests"
  type: "functional"
  base_url: "https://my.api.com/v1"
  tests:
    - name: "Get user by ID"
      method: "GET"
      endpoint: "/users/{id}"
      path_params:
        id: "123"
      expected:
        status: 200
Create a Gherkin feature in src/test/resources/features/my‑suite.feature:

gherkin
Feature: My suite

  Background:
    Given se carga la suite de pruebas "tests/my-suite.yaml"
    And se cargan los headers base

  @Functional
  Scenario: Get user by ID
    When se ejecuta la prueba "Get user by ID"
    Then el código de respuesta debe ser 200
    And el campo "id" debe ser "123"
Run:

bash
mvn test -Dcucumber.filter.tags="@Functional" -Dcucumber.filter.name="Get user by ID"
